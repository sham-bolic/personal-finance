# All of Piggy's writes require a UI confirmation gate, not just deletes

Piggy (the AI agent) is gaining write access to Goals, GoalContributions, and Budgets. We first considered gating only destructive writes (deletes) behind a UI confirmation, letting create/update/log-contribution execute directly on the theory that they're easily corrected after the fact. We reversed that: every agent-initiated write, including the additive `logGoalContribution`, now pauses as a staged Proposal and only executes after the user clicks an explicit Confirm button in the chat UI. Two reasons drove the reversal — the original ask was for confirmation before create _and_ remove, not remove alone; and a single uniform gate across all six write tools is a simpler mental model for the user (every agent write looks and behaves the same way) and a simpler invariant for the code (no tool is special-cased to skip staging).

We also considered, and rejected separately, having the agent describe the action in text and treat the user's next chat reply as consent, gated entirely by the LLM's own judgment of whether that reply means "yes." A misread reply ("yeah that guy") would still trigger a write with no deterministic gate. Instead, tool calls pause client-side (AI SDK's client-side-tool / `addToolResult` pattern) and only execute server-side after the user clicks Confirm/Cancel, rendered as a card in the chat.

This is harder to reverse later (it's load-bearing UI/tool-call architecture, not a prompt tweak), and a future reader might reasonably assume gating only deletes was "good enough" since creates/updates are correctable — worth recording why uniform gating was chosen instead.
